# PROGRESS — node-sync

Задача: добавить в Planshet синхронизацию с узлом хранилища по `docs/SYNC_SPEC.md`,
строго аддитивно к существующему приложению (пациенты, три теста, IndexedDB, экспорт CSV/ZIP).

Ветка: `feature/node-sync`.

## План

1. `tests/mock_node.py` — заглушка узла (stdlib), валидирует батч ровно так же, как настоящий
   `storage-core/webapp/routers/api_ingest.py` + `common/manifest.py`.
2. `index.html`: миграция БД v1 -> v2 (`settings`, `node_persons`, `outbox`), экран «Настройки»
   (+ QR), кэш пациентов узла и выбор из него, поля `timepoint`/`condition`, очередь `outbox`,
   фоновая отправка, бейдж и баннер в шапке.
3. Вендоренный QR-декодер (fallback к `BarcodeDetector`).
4. `tests/test.html` — asserts на чистом JS: формат `batch_id`, сборка батча, апгрейд БД, QR.
5. README: раздел «Синхронизация с узлом». Прогон сценариев §6 — сюда.

## Решения (и почему)

- **`batch_id`** — `<UTC yyyymmddThhmmssZ>_<8 hex>` (общий формат §A). Генерится **один раз** при
  постановке в outbox, хранится в записи очереди, все ретраи шлют тот же id. Узел валидирует
  `^\d{8}T\d{6}Z_[0-9a-z]{8}$` (`storage-core/common/manifest.py:19`); наши 8 hex — подмножество.
- **`test_id`** = `<test>_<локальный id записи>` (`words_12`, `motor_3`, `draw_7`). Уходит в
  `manifest.source_state.test_id`; `uid` строки реестра = `planshet:<device_id>:<test_id>`.
  `device_id` уходит в `manifest.machine`. В `batch_id` их нет — узел такой формат отвергает.
- **`manifest.files`** содержит `registry/records.jsonl` **и** файлы данных. Это не косметика:
  настоящий узел (`api_ingest.py:102-110`) требует, чтобы множество multipart-частей (кроме
  `manifest`) **точно совпадало** с `manifest.files`; лишняя или недостающая часть -> 422.
- **`manifest.created_at`** — время постановки батча в очередь. `created_at` строки реестра —
  время самого теста. Это разные вещи.
- **Очередь пополняется только при настроенном узле** (есть `node_url` и `api_key`). Иначе
  приложение вообще не трогает outbox — поведение без узла идентично текущему, и позднее
  подключение узла не выплёскивает на него всю историю разом. Для намеренного бэкфилла в
  «Настройках» есть кнопка «Поставить в очередь все локальные тесты».
- **Момент постановки в очередь.** «Моторика» и «Рисовалка» — сразу в `finish()` (запись готова).
  «Слова» — одна запись копит несколько букв/категорий, поэтому батч ставится в очередь при
  **уходе с экрана теста** (кнопка «Закончить», кнопка «назад», `pagehide`). Иначе фоновый цикл
  успел бы отправить запись с одной буквой, а вторая уже не доехала бы: `batch_id` тот же ->
  узел ответит `duplicate`, и данные потерялись бы молча.
- **Статус `error`** (сверх `pending|sent|auth_error` из спеки) — для 4xx, кроме 401/403
  (например 422 «bad manifest»). Такой батч не ретраится вечно: он виден в списке с текстом
  ошибки и кнопкой ручного повтора. Сеть и 5xx — как в спеке, остаются `pending`.
- **Порядок.** Отправка строго по возрастанию `id` очереди, по одному батчу за раз; первая
  сетевая ошибка обрывает цикл, чтобы не переставлять записи местами (сценарий 2).

## Проверка против живого узла (`https://localhost:8443`, 2026-07-10)

Узел поднят (Caddy -> uvicorn), сертификат в браузере уже доверенный: страница
`/setup/cert` открывается без предупреждения. Итог: **приложение не может достучаться до узла
из-за отсутствия CORS**, всё остальное в порядке.

| Запрос | Результат |
|---|---|
| `GET /setup/cert` (`mode:'no-cors'`, контроль сети и TLS) | доходит |
| `GET /api/ingest/status` без заголовков | `TypeError: Failed to fetch` (нет `Access-Control-Allow-Origin`) |
| `GET /api/ingest/status` с `X-Api-Key` | `Failed to fetch` (preflight) |
| `GET /api/persons` с `X-Api-Key` | `Failed to fetch` (preflight) |
| `POST /api/ingest/planshet` с `X-Api-Key` | `Failed to fetch` (preflight) |
| `curl -X OPTIONS /api/ingest/planshet` | `405`, заголовков `Access-Control-*` нет |
| `curl GET /api/persons` и `/api/ingest/status` без ключа | `401` (нужна сессия оператора) |

По итогам этой проверки в «Проверить связь» добавлено различение двух причин: и непринятый
сертификат, и запрет CORS дают одинаковый `Failed to fetch`, поэтому при ошибке приложение
делает контрольный `no-cors` запрос. Если тот доходит — сеть и TLS живы, значит дело в CORS,
и экран прямо это говорит, называя origin планшета. Если не доходит — прежняя подсказка про
сертификат. Оба ответа проверены на живом узле и на несуществующем адресе.

## РЕШЕНО 2026-07-10 (согласовано с агентом storage-core, его N19–N20)

Три вопроса ниже закрыты. Итог — Planshet-сторона стала **проще**, узел-сторона — минимальна.

1. **Справочник пациентов (`/api/persons`) убран из Planshet.** Базу пациентов наружу решили
   не отдавать. Пациента заводит оператор вручную (ФИО/код), запись помечается
   `patient_ref:{local_name}` — это теперь **штатный и единственный путь**, а не fallback.
   Сшивку с человеком делает identity-метод узла (`ingester/adapters/planshet.py`, ветка
   `local_name`), она на узле уже построена. На Planshet удалено: fetch `/api/persons`, store
   `node_persons`, UI выбора из списка узла, поле `nodePerson` у пациента, ветка
   `patient_ref:{node_person_id}`. SYNC_SPEC §3 переписан. Изменений на узле по persons — ноль.
2. **«Проверить связь» и 401 на `/api/ingest/status`.** Эндпоинт закрыт сессией оператора,
   ключу отдаёт 401 — это НЕ ошибка, а по SYNC_SPEC §2 («ожидать 200/401») значит «связь есть».
   Приложение так и трактует: показывает «Связь с узлом есть», поясняет 401 и что сам ключ
   проверится при первой отправке. Статус-борд ключу узел намеренно не отдаёт.
3. **CORS добавлен на узле** (`CORSMiddleware`, allowlist из новой переменной
   `PLANSHET_CORS_ORIGINS`, пусто по умолчанию — поведение узла не меняется; не wildcard, без
   cookie). На Planshet ничего чинить не нужно: «Проверить связь» уже различает CORS-блок и
   непринятый сертификат (контрольный `no-cors` запрос) и подсказывает добавить origin в
   `PLANSHET_CORS_ORIGINS`. Проверено на живом узле: до правки CORS приложение видит блок и
   называет origin; узел-агент подтвердил, что с разрешённым origin приходит `200`+`ACAO`, а
   401 на статусе — тоже с CORS-заголовком (успешный fetch, не «Failed to fetch»).

Для запуска связки оператору осталось: узнать origin, с которого открыт САМ планшет (адрес в
строке браузера планшета, не адрес узла), прописать в `.env` узла
`PLANSHET_CORS_ORIGINS=<этот origin>` и `docker compose up -d webapp`.

## Вопросы к контрактам (storage-core) — исходные формулировки

1. **`GET /api/persons` не принимает `X-Api-Key`.** SYNC_SPEC §3 требует ходить туда с ключом
   планшета, но в `storage-core/webapp/routers/api_persons.py:15,24` зависимость —
   `auth.require_role("researcher")`, то есть сессионная кука оператора. Ключ со scope
   `ingest:planshet` там получит 401. Нужен либо scope `read:persons` для api-ключей, либо
   отдельный эндпоинт. Заглушка `tests/mock_node.py` реализует поведение **по спеке** (ключ
   принимается), поэтому сценарии проходят; против настоящего узла справочник пациентов
   сейчас не заработает.
2. **`GET /api/ingest/status` — то же самое** (`api_ingest.py:45-47`, `require_role("researcher")`).
   Для «Проверить связь» это терпимо: спека разрешает считать 401 успехом (узел ответил, TLS жив),
   и приложение так и делает — различает «узел отвечает» и «сети/сертификата нет».
3. **CORS на узле отсутствует — подтверждено на живом узле.** Ни middleware в `webapp/main.py`,
   ни заголовков в `Caddyfile`; `OPTIONS /api/ingest/planshet` отвечает `405` без `Access-Control-*`.
   Пока планшетное приложение открыто не с самого узла, браузер блокирует и `GET /api/persons`,
   и `POST /api/ingest/planshet`. SYNC_SPEC §5 утверждает, что «узел разрешает origin планшета
   (уже учтено на стороне узла)» — это пока не так. Прокси не выдумывал: заглушка отвечает на
   `OPTIONS` и шлёт `Access-Control-Allow-*`, как спека обещает от узла. Варианты решения на
   стороне storage-core: `CORSMiddleware` для origin планшета, заголовки в `Caddyfile`, либо
   раздача самого приложения с узла (тогда origin один и CORS не нужен).

## Прогон

### tests/test.html — 53/53 зелёных (Chrome, Windows)

Страница читает `../index.html`, исполняет его код и проверяет **сам продукт**, а не копию.
Запуск: любой статический сервер из корня репозитория, затем `/tests/test.html`.

- `batch_id`: регексп в приложении совпадает с узловым; 200 сгенерированных id проходят
  валидацию; время в UTC; device_id/test_id внутрь id не попадают.
- `buildBatch` для всех трёх тестов: поля манифеста, `source_state.test_id`, `machine`,
  `uid`, оба вида `patient_ref`, `timepoint`/`condition`, `manifest.created_at` != `created_at`
  записи, аудио только у пунктов с записью, траектория без BOM.
- **Провод**: `FormData` заворачивается в `Request` и разбирается как настоящий multipart —
  множество частей равно `manifest.files`, sha256 и size каждой части сходятся, `records.jsonl`
  парсится. Это ровно те проверки, которые делает узел (`api_ingest.py:102-119`).
- Апгрейд БД v1 -> v2: новые сторы созданы, индексы `outbox` на месте, старые данные и индекс
  `patientId` не потеряны.
- QR: таблицы ECC сверены с ёмкостями из ISO/IEC 18004 (v1L=19 … v40H=1276) — это независимая
  проверка, а не самосогласованность с нашим кодировщиком; Рид-Соломон чинит ec/2 ошибок и
  честно возвращает null при переизбытке; полный круг кодирование -> картинка -> декодирование
  на 4 уровнях ECC x 8 масках, на версиях 1…26, с поворотом 7° и неравномерным масштабом,
  с грязью на изображении.

### Сценарии SYNC_SPEC §6 — прогнаны в браузере, НЕ против mock_node.py

На этой машине нет Python (только Store-заглушка `python3.exe`), нет Node, докер-движок не
поднят — `tests/mock_node.py` здесь запустить нечем. Поэтому сценарии прогонялись по реальному
UI приложения (клики по кнопкам, а не вызовы внутренних функций) против заглушки на уровне
`fetch`, которая **валидирует батч теми же правилами**, что и `mock_node.py`: формат `batch_id`,
совпадение частей с `manifest.files`, sha256 и size каждой части, разбор `records.jsonl`.
За семь отправленных батчей — ноль замечаний валидатора.

| Сценарий §6 | Результат |
|---|---|
| 1. Онлайн: тест -> корректный батч | 3 батча приняты, валидатор молчит, бейдж скрылся |
| 2. Офлайн: 3 теста -> 3 pending -> узел вернулся | все доехали, **порядок batch_id сохранён** |
| 3. Узел ответил `duplicate` | статус `sent`, в очереди пометка «duplicate», дублей на узле нет |
| 4. Узел ответил 401 | статус `auth_error`, баннер «проверь ключ», бейдж жёлтый, **ретраев нет**; смена ключа в настройках вернула запись в очередь и она ушла |

Дополнительно проверено на живом UI:
- повторная отправка уже отправленной записи шлёт **тот же** `batch_id` -> узел говорит
  `duplicate` -> запись остаётся `sent`;
- «Слова»: две буквы в одну запись; батч встаёт в очередь **только при уходе с экрана** и несёт
  `files/audio_1.webm` + `files/audio_2.webm`;
- «Рисовалка»: батч несёт `files/drawing.png` + `files/trajectory.csv`;
- выбор пациента из кэша узла даёт `patient_ref.node_person_id`; «Точка»/«Нагрузка» доезжают
  в записи реестра (`timepoint:2`, `condition:'pre_load'`).

**Осталось руками** (нужна машина с Python) — чек-лист в README, раздел «Синхронизация с узлом»:
поднять `tests/mock_node.py`, повторить те же 4 сценария и убедиться, что в `tests/_batches/`
лежат распакованные батчи с `manifest.json` и `_complete`.

### Без узла приложение неотличимо от прежнего

Проверено в браузере: пустые настройки -> ни одного сетевого запроса после старта, бейдж и
баннер скрыты, главный экран показывает обычную форму добавления пациента, в консоли пусто.
Очередь при ненастроенном узле не пополняется вовсе.

## Известные ограничения

- Вендоренный QR-декодер проверен полным кругом против собственного кодировщика и на
  геометрических искажениях, но **не против QR, снятого камерой с экрана узла** — на планшете
  сработает `BarcodeDetector`, вендоренный путь остаётся для десктопных браузеров.
- `pagehide` вызывает постановку «Слов» в очередь асинхронно: если вкладку убить мгновенно,
  запись останется локально. Лечится кнопкой «Поставить в очередь все локальные тесты».
